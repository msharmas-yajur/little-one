Little One — Product Plan & Architecture

A gentle, growing storybook for children from about six months to five years — a shared lap activity (a grown-up reads aloud, the little one taps along) that designs and improves itself under human review. This document is the orientation for anyone building on the project: the vision, the architecture, and — the point of this update — what is actually delivered.

Prepared 5 July 2026 · Substantially updated 7 July 2026 (three PRDs; the Generative Arc largely built — voice now live) · Source of truth: the project knowledge base (knowledge-base/) · No child data appears anywhere in this document.

Read this in five lines

  • The app is a fixed engine + reviewed data. Behaviour changes are data changes — versioned, diffable, reversible. There is no LLM on the child's device.
  • It improves itself at the development level: an automated Claude session proposes one improvement at a time as a pull request; a human gate decides; the next run learns from that decision.
  • It now generates its own experiences — content, and the interactions built on them — as reviewable, capability-bounded data + code, judged by an automated panel of six expert "seats" and gated by a human.
  • Privacy is structural: generation is global (from the public knowledge base); personalisation is local (on-device, no LLM); only anonymised, aggregate signals ever leave.
  • It is a deliberate rehearsal of guidance-driven workflow change (e.g. clinical protocols): guidance update → reviewed diff → conformance checks → audit trail → rollback.

② The promises the whole plan is built on

  • Privacy by construction. A child's profile and interactions live only on the family's device — never uploaded, never in the repository. No names are ever collected. Only anonymised, aggregate signals may ever leave.
  • The grown-up stays in the loop. This is a lap activity; the app supports the adult, it never replaces them. All guidance is a practice for the adult, not a curriculum for the child.
  • Forgiving by design. No scores to lose, no fail states, no scolding — a wrong tap gets a gentle nudge.
  • Evidence-based. Content and pacing trace to developmental science (Gopnik, Galinsky, Kuhl, Byers-Heinlein, Faber & King) recorded as sourced notes in the knowledge base.
  • Everything reviewable. The engine is fixed; behaviour changes are data — versioned, diffable, reversible. Nothing generated reaches a child unreviewed.

③ The architecture — one mental model

The whole system is three layers and one loop. Hold these and everything else follows.

1 · A fixed engineA small static site (no backend) that renders declarative data. It never changes to add an experience — it reads a bounded schema and draws what the data says.
2 · Reviewed dataStories, games, interaction mechanics, the developmental knowledge base, and every policy live as versioned files. This is where the app's "behaviour" and its "learning" actually live.
3 · On-device personalisationA local, interpretable model of the parent-and-child pair selects and paces from the data. No machine learning on child signals, no network, no LLM on the device.
The loop. At the development level, an automated Claude session reads the knowledge base and proposes an improvement — new content, or a new interaction mechanic — as a pull request. Automated checks and a panel of judges vet it; a human merges or closes; the next run reads that decision. The engine's weights never change; the reviewed data it reads changes. (This is the CLHF / in-context-learning pattern, adopted from CopilotKit's self-improving-agents idea — the pattern, not the runtime library.)
Generation is global; personalisation is local; the feedback that connects them is anonymised and aggregate. Generation happens at the development level from the public knowledge base, so it touches no child data by construction. The on-device profile only chooses from the shelf — it never phones the publisher. The only data that ever leaves a device is opt-in, anonymised, threshold-gated cohort signals (never a raw per-child trace); true per-child learning is reserved for a family's own bring-your-own-key instance.

④ The gate stack — how autonomous generation stays safe

The reason an LLM can be allowed to propose content and interaction code is that nothing it produces reaches a child unless it survives every layer below. Each layer is independent; the deterministic ones cannot be overridden by the judgemental ones.

Grammar bound (content). The experience schema is additionalProperties:false over a closed vocabulary — generated content cannot express a network call, storage, or script, because no such field exists.
Capability denylist (code). For a new mechanic that adds engine code, a linter scans the added JavaScript for forbidden capabilities (fetch, sockets, storage, eval, external URLs, <script>) and enforces additive-and-small. The code equivalent of the grammar bound.
Deterministic checks. No names/PII, valid tags and pictures, no banned framing, schema-conformant, valid JavaScript, the site still builds. Any red check blocks the proposal.
Scope allowlist. A content proposal may touch only content + the knowledge base. A mechanic proposal may touch the schema, the engine renderer, the layout, content, and docs — never its own validators or charters; it cannot weaken its own gate.
The Review Council. Six independent expert seats judge the artifact against sourced rubrics; a Chairman synthesises. Unanimous to publish; any rejection or the Guardian's hold escalates to a human.
The human gate. A person merges (it ships) or closes (a recorded "no"). The next run reads that decision. Classes (new kinds of experience) are always human-approved.
Charter-as-data. Every autonomous actor's own instructions — the Gardener, the Mechanic, the Council — are versioned files (GARDENER.md, MECHANIC.md, COUNCIL.md), changed only through the same reviewed process they govern. Changing the process uses the process.

PRD 0001 — The Adaptive Arc v1 LIVE

Make the experience adapt to this child and this family — privately, gently, explainably.

The four pillars (all decided, all built)

🕊 Passive-first signalsQuiet observation (opens, repeats, completions). A one-tap "noticing question" only when there's a real hypothesis — at most one per session, two per week.
🌱 Autonomy is a lifecycleDiscovery (explore variety) → Calibration (gentle questions begin) → Partnership ("adapt automatically" or "ask me first").
👨‍👧 The dyad, not just the childThe unit is the parent-and-child pair, with up to two language/culture tracks. The app is never the language teacher — the grown-up's voice is the mechanism (Kuhl's social gating).
🧭 Skills as vocabulary, never a report cardContent is tagged with Galinsky's seven life skills and age bands; selection reasons in those terms. Moments noticed, not scores.

In the app today

  • A "Make it hers" setup asks only birth month and home languages (chosen from a pill list) — never a name.
  • The child page leads with a big "Next for…" card and a per-child menu ordering — nothing is ever hidden.
  • Session-end noticing questions change what is served next; profile export/import; airplane-mode works; zero network requests carry child data.
  • Mobile full-screen (installable), no-scroll question pages, playful per-story music, a warm reinforcing voice, lift-the-flap and camera-mirror peek-a-boo.

PRD 0002 — The Gardener Loop LIVE · proven

A self-improving application whose learning lives in its knowledge base: an automated session proposes one reviewed improvement at a time, and learns from which proposals the owner accepts.

One pass

Trigger. A person presses "Run Gardener" (manual; $0 until pressed). Never triggered by outside input — untrusted text must not steer an automated change.
Gather. It reads only owner-committed files: the knowledge base, the content library and its tags, a computed coverage map (age band × seven skills), and the fate of its own past proposals.
Reason. The single highest-value action: ingest a research note, or fill the biggest content gap with one new story/game, or a small consistency fix. Doing nothing is valid.
Gate. Deterministic checks + scope allowlist (content & knowledge-base only). A failing check opens no PR.
Human gate. Merge (ship) or close (recorded "no"). The next run reads that fate — that is the learning.
Proven in production. Its first two autonomous proposals — a story for the one life-skill with no content, and a story for the youngest age band — were reviewed, merged, and are live. It read its own merged history on the next run.

The Gardener has since graduated to also propose new interaction mechanics — see PRD 0003 → The Mechanic loop.

PRD 0003 — The Generative Arc BUILDING · most delivered

From static pre-generated pages to generated experiences — stories, the interactions built on them, a warm voice, and later music — each one reviewed on the go by the panel, with the human moving from gatekeeper-of-everything to approver of kinds of experience and auditor of samples.

The principle that unlocks it

The app is a digital replica of a physical book. A book never knows the child — the lap does. So the generative pipeline runs on the public knowledge base and touches no child data by construction; the on-device profile only composes and paces from what was generated.

The three planes of generation

📖 Stories & games LIVEThe Gardener writes them; this PRD scales them from one-at-a-time toward reviewed pools.
🧩 Interactions — generative UI LIVEThe engine is a fixed renderer; every interaction is data in a bounded schema. A new interaction arrives as reviewable data + a small renderer, not scattered code.
🎙 Voice LIVEApproved phrases rendered once into small static clips (Sarvam) — no robotic device voice, works offline; a settings picker and per-story voices. Music followed the same idea (generative synth).

Delivered: the experience schema & the schema-driven engine LIVE

Today's mechanics are extracted into a capability-bounded declarative schema. The safety is grammatical: because every object is a closed field set, generated UI cannot say anything unsafe. The engine became a pure renderer — a type registry where adding a new interaction is one schema branch + one registry entry + example content, with no bespoke engine surgery and every existing experience byte-identical.

Delivered: the Review Council BUILT

Six independent seats review each artifact in parallel, each bound to specific knowledge-base pages (its "constitution") and required to cite the principle that passes or fails; a fixed Chairman synthesises; the publish/revise/escalate decision is computed mechanically from the verdicts (the model never overrides the gate).

🔭 Explorer's Seat (after Gopnik) — open-ended exploration, or drilling?
🧭 Skills Seat (after Galinsky) — does it truly exercise the life-skill it claims?
💬 Voice Seat (after Kuhl; Byers-Heinlein) — grown-up still the language teacher? serve-and-return?
🕊 Tone Seat (after Faber & King) — descriptive, feelings first, never scolds or tests.
🛡 Guardian Seat — privacy, no fail states, schema bounds; the strictest seat, backed by machine checks no approval can override.
🤱 Lap Seat — the family's advocate: readable aloud at 9pm by a tired grown-up with a squirming one-year-old?
It already caught a real defect. On its first live review the Guardian Seat flagged that a story referred to the child as "her/she", violating the project's own neutral-referent rule — a bug two earlier passes had missed. The Skills Seat separately found a skill tag that wasn't truly exercised, and the Explorer and Tone seats disagreed on one line, which the Chairman surfaced for the human. Panel disagreement is signal, not noise.

Grounded in the LLM-as-a-Judge literature and Karpathy's llm-council: rubric-anchored verdicts (not "which reads nicer"), adversarial charters, the immovable deterministic floor, and a calibration loop where a human's overturn becomes a teaching example in the offending seat's charter.

Delivered: the Mechanic loop — the Gardener graduates BUILT · proven

The Gardener now also proposes a new interaction mechanic (a class): a schema branch + an additive engine renderer + serialisation + one example, as one always-human-gated pull request. This is the one place an LLM edits engine code, so it carries an extra bound — the capability denylist above — plus a scope allowlist that forbids it from editing its own validators or charters.

Proof, unattended. Its first run autonomously designed a counting game — "How many? Tap each one" — writing the schema branch, the renderer (using only existing engine primitives — no network, no storage, no eval), the serialisation, and a first-of-its-kind 3-4 year example. It passed every gate, and opened a human-gated pull request for review. Independently re-verified: capability-clean, valid, schema-conformant. A machine proposing a new kind of interaction as reviewable, bounded code — and a human still decides.

How the gate graduates (classes vs instances)

Classes are approved by a human, once. A new interaction mechanic, a new voice, a new template — approved before any instance exists.
Instances are approved by the Council. Variants within an approved kind pass machine checks, then the six seats (unanimous to publish; any rejection or Guardian hold → human; one revision cycle).
Humans audit samples. Every flagged item, every first-of-its-kind, and a random sample — and every overturn teaches the offending seat. Its agreement rate with the owner earns it more trust over time.

Delivered: the warm voice LIVE

Provider: Sarvam AI — chosen over the obvious Western option because it is Indian-language-first (warm Indian-accent English plus Hindi, Bengali, Tamil, Telugu and more), matching the app's home-language list and opening the door to bilingual voice (the same line in English and the home language). At build time, each approved phrase is rendered once into a small static clip; the player plays the clip when one exists for the active voice and falls back to the device voice otherwise, so coverage grows safely over time.

Two things you asked for, both shipped. A voice picker in settings — Auto (per story) · Ritu (default) · Kavya · Priya · Pooja · Shreya — expressive, playful voices (Sarvam bulbul:v3), each with a live preview. And different voices for different stories and games — a story or game can carry its own voice (e.g. Peek-a-boo in one voice, Splish Splash in another). Precedence: the family's picked voice › the content's own voice › the default. Re-rendering runs in CI from a stored key and opens a pull request; the key never leaves CI.

⑧ Why this matters beyond a storybook

The self-improving loop is a deliberate rehearsal of a hard, general problem: workflows that must change when official guidance changes — most sharply, clinical protocols. The mapping is exact, and it is the most transferable part of this work for teams building other solutions.

In Little OneIn a guidance-driven workflow
Research note ingested → proposal as a reviewable diff, with provenanceProtocol update → exact pathway change, visible before it applies
Schema grammar + capability denylist bound what can be generatedConformance / terminology / formulary rules bound what can be ordered
Panel of judges + a human gate; classes always human-approvedCommittee review (P&T / IRB); standing orders vs case-by-case
Anonymised cohort signals improve the next generationDe-identified outcomes registry improves the next protocol
Merge/close decisions feed the next run (institutional memory)Governance feedback becomes durable policy
Git history · one-click revertAudit trail · rollback

The lesson teams can lift directly: put the changing knowledge in a reviewed data layer, bound generation by grammar and capability (not trust), separate classes from instances, and make the human's decision a first-class training signal.

⑨ What comes after

  • Bilingual voice — the warm voice is live; next, narrate the same line in English and the home language on one page (Sarvam covers ten Indic languages), and widen clip coverage beyond the current phrase set.
  • Variant pools + aggregate telemetry — the Gardener generates reviewed pools; the device composes per child locally; anonymised cohort signals steer the next generation.
  • Council upgrades — best-of-N selection among variants; the calibration loop (owner overturns → seat teaching examples); seats on different AI models for the highest-stakes reviews.
  • Ages 2–5 arcs — the tagging, coverage map, and older-band mechanics (odd-one-out, counting) already open the door; more colours / opposites / feelings mechanics follow.
  • Bring-your-own-key — a family's own instance that learns per child on their own key and data; and profile portability through the family's own cloud account. We never operate an identity service.
  • Gardener on a schedule — after enough trusted manual runs.
